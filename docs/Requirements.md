# 要求

- コミットの履歴がかわいくて見やすい感じ
- クローンが楽にできる
- ブランチわかりやすいといい
- 差分も簡単に見れるといい
- 画面内のメッセージはすべて日本語で表示する
- エラーメッセージ、説明文、ボタン名、確認文言も日本語に統一する

## クローン

- メイン画面を開いて既存のフォルダのパスを入力するとそのフォルダのリポジトリを一覧で見れる
- クローンボタン押したらダイアログ開く
- ユーザーはクローン元のリポジトリURLを入力できる
- クリップボードの文字列をURLの初期値にする
- ユーザーは任意のフォルダ名を入力できる（リポジトリ名を初期値とする）
- 実行ボタンを押すと、最初に入力したパスのフォルダの下に新しくフォルダ作ってクローンができる


# Git GUIツール提案 — 相談資料

## この資料について

新しいGit GUIツールを作りたいと考えています。アイデアの方向性について、Gitに詳しいあなたの意見を聞かせてください。まだ決まっていない論点もあるので、一緒に詰めてもらえると助かります。

---

## 作りたい理由

私自身、Gitを理解して使えるようになりたいと思いながら何度か挫折し、今も「なんとなく使っている」状態です。コマンドを叩けば動くし、簡単な作業はできる。でも、ちょっと踏み込んだ操作になると途端に怖くなる。`reset` と `revert` の使い分けで毎回ドキッとするし、`HEAD` が何を指しているのかを毎回確認しないと不安になる。

既存のGUIツール(SourceTree, GitHub Desktopなど)も試しました。操作は楽になるのですが、**Git自体の理解は深まりませんでした**。むしろボタンを押して結果だけ見ている感じで、内部で何が起きているのかがますます見えなくなる感覚すらありました。

なので、自分のようなユーザーのために「**使いながらGitが分かっていく**」ツールを作りたいと思っています。そして同じくらい大事なのが、**熟練者になっても使い続けられる**こと。初心者向けに用語を言い換えて隠してしまうツールだと、熟練するにつれて「本物のGit用語」との断絶が生まれてしまい、結局ツールを卒業しなければいけなくなります。そうではなく、初心者から熟練者までずっと同じ道具で歩ける設計にしたい。

---

## コンセプト

> **Gitの概念や用語を視覚的に表し、わかりやすくするGUIツール**

これが一行で言えるコンセプトです。「操作を楽にする」ためのツールではなく、「**理解の解像度を上げる**」ためのツールです。

### 2つの設計原則

1. **隠さない、説明する** — Gitの正式用語(`commit`, `stage`, `HEAD`, `rebase` など)はそのまま表示する。代わりに、控えめな補足説明を添える。熟練するにつれて補足を読まなくなる、という自然な卒業の形を目指す。

2. **状態を常に見せる** — Gitが難しい大きな理由は「今ファイルがどこにあるのか」「今どのブランチにいるのか」といった状態が目に見えないこと。このツールでは状態を常時、視覚的に見せる。

---

## ターゲットユーザー

- **メインターゲット**: Gitを理解して使いこなしたいが挫折した人、コマンドは少し触ったが怖くなった人、なんとなく使っている人
- **サブターゲット**: 熟練者が日常的に使っても困らないこと。むしろ細かい機能(行単位のステージング、直接コマンド入力など)も備える

---


## 「なぜ?」ボタン

すべての操作ボタンの横に小さな `?` マークを置きます。クリックすると、そのコマンドの**目的**(whatではなくwhy)が1〜2文で表示されます。

例:
- **Stagingエリアの `?`** → 「一度に全部コミットせず、関連する変更だけ選んで記録できるようにするための控え室です」
- **`git reset` の `?`** → 「`--soft / --mixed / --hard` の違い」のような、熟練者がたまに忘れるリファレンスも入れる

初心者向けの機能のつもりですが、熟練者にとっても「たまに忘れる詳細」のリファレンスとして機能するはずです。一つの画面要素が世代を超えて役立つ設計にしたい。

---

## 用語の扱いについて(重要な判断)

初心者向けツールだと「コミット」を「保存する」と言い換えたくなりますが、**このツールではそれをやりません**。理由は:

1. 言い換えると、熟練する過程で「本物の用語」との接続が切れてしまう
2. 熟練者が使うときに迷う
3. そもそも「Gitの概念を理解する」ことが目的なので、用語から逃げたら本末転倒

代わりに、**正式名称はそのまま、控えめな補足を添える**スタイルにします。たとえばエリアのヘッダーはこう:

```
Staging
コミット予定
```

"Staging" が主役で、下の「コミット予定」はあくまで補足。補足は薄いグレーで小さく。熟練者は上だけ見ればいいし、初心者は下も読む。使っているうちに下を読まなくなる ── これが自然な学習曲線だと考えています。

---

## 想定する差別化

- **SourceTree / GitKraken** — 熟練者向けに機能は充実しているが、初心者が使ってもGit自体の理解は深まらない。このツールは理解を目的とする
- **GitHub Desktop** — 初心者向けにシンプルだが、用語を隠しているためGitの概念が学べない。熟練すると物足りなくなり、結局別ツールに移行が必要
- **学習サイト (Learn Git Branching 等)** — 概念の学習には良いが、実際の作業には使えない。このツールは学習と実作業を同じ画面で行う

一言でいうと「**実作業しながら概念が身についていく**」というポジションを狙いたいです。
